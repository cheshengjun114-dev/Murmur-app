# Murmur 프로젝트 기획서

## 1. 제품 개요

### 제품 이름

Murmur

### 한 줄 소개

20~30대가 익명으로 고민, 일상, 썰을 공유하고 공감할 수 있는 커뮤니티 플랫폼

### 만들게 된 이유

기존 SNS는 팔로워, 좋아요 수 등 비교와 과시 중심 문화가 강해 부담을 느끼는 경우가 많습니다.
그래서 사용자가 익명으로 편하게 글을 작성하고, 공감과 댓글을 통해 소통할 수 있는 커뮤니티 서비스를 만들고자 했습니다.

또한 단순 게시판이 아니라 신고, 블라인드 처리, 인기글 기능 등을 포함해 실제 서비스 운영 흐름까지 고려한 프로젝트를 목표로 합니다.

## 2. 타겟 사용자

### 주요 사용자

- 20~30대 대학생
- 취업 준비생
- 직장인
- 익명으로 일상이나 고민을 공유하고 싶은 사용자

### 사용자가 원하는 것

- 부담 없이 익명으로 글을 작성하고 싶다.
- 회사, 취업, 연애, 인간관계 고민을 편하게 공유하고 싶다.
- 다른 사람들의 공감이나 댓글 반응을 받고 싶다.
- 인기글이나 썰을 보며 재미를 느끼고 싶다.
- 기존 SNS보다 비교와 과시가 적은 커뮤니티를 원한다.

## 3. 핵심 기능

### 높음

- 회원가입 / 로그인: JWT 기반 사용자 인증
- 게시글 CRUD: 익명 게시글 작성, 수정, 삭제, 조회
- 댓글 / 대댓글: 게시글에 댓글 및 답글 작성
- 반응 시스템: 공감해요, ㅋㅋㅋ, 힘내요 반응 기능
- 신고 / 블라인드: 신고 누적 시 게시글 자동 숨김 처리

### 중간

- 인기글 기능: 조회수, 댓글수, 반응수를 기반으로 인기글 제공
- 익명 번호 시스템: 게시글 단위 익명1, 익명2 형태로 표시
- 카테고리 검색: 회사, 연애, 고민 등 카테고리별 조회

### 낮음

- 북마크 기능: 게시글 저장 기능
- 관리자 페이지: 신고 게시글 관리 기능

## 3-1. 강사 피드백 반영 정책

### 익명성과 관리자 식별 범위

- 일반 사용자는 게시글/댓글 작성자의 실제 계정 정보를 볼 수 없습니다.
- 관리자는 운영 정책 위반, 신고 처리, 제재 판단을 위해 작성자의 내부 사용자 ID를 확인할 수 있습니다.
- 즉, Murmur의 익명성은 "사용자 간 익명"이며, "운영자에게도 완전 익명"인 서비스는 아닙니다.
- 관리자의 작성자 조회는 관리자 권한 API에서만 허용하고, 조회 로그를 남기는 방향으로 설계합니다.

### 익명 번호 정책

- 익명 번호는 게시글 상세 화면 단위로 부여합니다.
- 게시글 작성자는 항상 `익명1(글쓴이)`로 표시합니다.
- 댓글 작성자와 대댓글 작성자는 해당 게시글 안에서 처음 등장한 순서대로 `익명2`, `익명3`처럼 부여합니다.
- 같은 사용자가 같은 게시글 안에서 댓글이나 대댓글을 여러 번 작성하면 항상 같은 익명 번호를 유지합니다.
- 게시글 작성자가 댓글이나 대댓글을 작성하면 `익명1(글쓴이)`로 표시합니다.
- 다른 게시글로 이동하면 익명 번호는 다시 새로 계산합니다.

### 신고 및 블라인드 정책

- 로그인한 사용자만 신고할 수 있습니다.
- 자기 글과 자기 댓글은 신고할 수 없습니다.
- 같은 사용자는 같은 대상에 대해 1회만 신고할 수 있습니다.
- 신고 대상은 게시글, 댓글, 대댓글로 확장 가능하게 설계합니다.
- 게시글은 신고 5회 이상 누적 시 자동 숨김 처리합니다.
- 댓글/대댓글은 신고 3회 이상 누적 시 자동 숨김 처리합니다.
- 자동 숨김 처리된 콘텐츠는 관리자 페이지에서 확인, 해제, 유지, 삭제 처리할 수 있습니다.
- 숨김 해제 시 신고 횟수는 기록으로 남기되, 같은 신고 누적으로 즉시 다시 숨김 처리되지 않도록 관리자 해제 상태를 별도로 둡니다.
- 자동 숨김만으로 작성자에게 즉시 페널티를 부여하지 않습니다.
- 관리자가 신고 내용을 검토한 뒤 경고, 일정 기간 정지, 영구 정지 중 하나를 결정합니다.

### 회원가입이 필요한 이유

- 익명 서비스여도 게시글 수정/삭제 권한 확인이 필요합니다.
- 신고 남용 방지, 중복 신고 차단, 제재 처리, 관리자 운영을 위해 계정 식별이 필요합니다.
- 북마크, 마이페이지, 내가 쓴 글 관리 같은 개인 기능을 제공하기 위해 회원가입이 필요합니다.
- 사용자는 서로를 알 수 없지만, 서버는 권한 검증을 위해 작성자 ID를 저장합니다.

### 같은 사람이 쓴 글 추적 범위

- 일반 사용자는 같은 사람이 쓴 다른 게시글을 추적할 수 없습니다.
- 관리자는 신고/제재 목적에 한해 특정 계정이 작성한 게시글과 댓글을 조회할 수 있습니다.
- 공개 화면에서는 게시글마다 익명 번호가 새로 부여되므로, 서로 다른 게시글의 작성자가 같은 사람인지 알 수 없습니다.

### 익명 글 수정/삭제 방식

- 게시글, 댓글, 대댓글 테이블에는 실제 작성자 `userId`를 저장합니다.
- 화면에는 `userId`를 노출하지 않고 익명 번호만 표시합니다.
- 수정/삭제 요청이 들어오면 JWT에서 꺼낸 로그인 사용자 ID와 콘텐츠의 작성자 ID를 서버에서 비교합니다.
- 두 ID가 같으면 수정/삭제를 허용하고, 다르면 403 Forbidden 응답을 반환합니다.

### Redis 사용 이유

- Refresh Token을 Redis에 저장해 로그아웃, 토큰 재발급, 강제 로그아웃을 제어합니다.
- Redis TTL을 이용하면 Refresh Token 만료 시간을 자동으로 관리할 수 있습니다.
- 추후 인기글 점수, 조회수 증가량, 신고 카운트 캐싱처럼 자주 읽고 쓰는 데이터를 빠르게 처리하는 데 활용할 수 있습니다.
- MySQL은 영구 데이터 저장소, Redis는 빠른 만료/캐시/인증 보조 저장소 역할로 분리합니다.
