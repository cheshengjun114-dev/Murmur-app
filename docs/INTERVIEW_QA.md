# Murmur 면접 대비 Q&A

## Q1. 익명 서비스인데 회원가입은 왜 받았나요?

익명은 사용자끼리 서로의 실제 계정을 모르게 하는 정책입니다.
하지만 서버는 권한 검증을 해야 합니다.

회원가입이 필요한 이유는 다음과 같습니다.

- 본인이 쓴 글만 수정/삭제할 수 있게 하기 위해
- 신고 남용과 중복 신고를 막기 위해
- 운영 정책 위반 사용자를 제재하기 위해
- 북마크, 마이페이지 같은 개인 기능을 제공하기 위해
- Refresh Token을 사용자 단위로 관리하기 위해

즉, Murmur의 익명성은 사용자 간 익명이며, 서버 내부에서는 권한 관리를 위해 사용자 ID를 저장합니다.

## Q2. 관리자도 실제 작성자가 누구인지 알 수 없나요?

관리자는 알 수 있습니다.

일반 사용자는 게시글과 댓글 작성자의 실제 계정을 볼 수 없습니다.
하지만 관리자는 신고 처리와 제재 판단을 위해 작성자 내부 ID, 이메일, 닉네임을 확인할 수 있습니다.

이 설계는 완전 익명 서비스가 아니라 운영 가능한 익명 커뮤니티를 목표로 한 설계입니다.

## Q3. 익명 번호는 어떻게 유지되나요?

익명 번호는 게시글 상세 화면 단위로 유지합니다.

- 게시글 작성자는 항상 `익명1(글쓴이)`입니다.
- 댓글/대댓글 작성자는 해당 게시글 안에서 처음 등장한 순서대로 `익명2`, `익명3`처럼 부여됩니다.
- 같은 사용자가 같은 게시글 안에서 여러 번 댓글을 쓰면 같은 익명 번호를 유지합니다.
- 다른 게시글로 이동하면 익명 번호는 새로 부여됩니다.

이를 위해 `anonymous_mappings` 테이블에 게시글 ID, 사용자 ID, 익명 번호를 저장합니다.

## Q4. 익명인데 본인이 쓴 글 수정/삭제는 어떻게 하나요?

화면에는 사용자 ID를 노출하지 않지만, 서버 DB에는 게시글 작성자 ID를 저장합니다.

수정/삭제 요청이 들어오면 JWT에서 로그인 사용자 ID를 꺼내고, 게시글의 작성자 ID와 비교합니다.

- 같으면 수정/삭제 허용
- 다르면 `403 Forbidden`

이 방식으로 익명성을 유지하면서도 권한 검증을 할 수 있습니다.

## Q5. 같은 사람이 쓴 글을 일반 사용자가 추적할 수 있나요?

일반 사용자는 추적할 수 없습니다.

익명 번호는 게시글마다 새로 부여되기 때문에, 서로 다른 게시글의 `익명1`이 같은 사람인지 알 수 없습니다.
관리자는 신고/제재 목적에 한해 내부 사용자 ID 기준으로 작성 글과 댓글을 추적할 수 있습니다.

## Q6. 신고 정책은 어떻게 설계했나요?

현재 정책은 다음과 같습니다.

- 로그인한 사용자만 신고 가능
- 본인 게시글 신고 불가
- 같은 사용자는 같은 게시글을 한 번만 신고 가능
- 게시글 신고 5회 이상이면 자동 블라인드
- 관리자는 블라인드 글을 확인하고 숨김 해제, 유지, 삭제 처리 가능
- 자동 블라인드만으로 즉시 페널티를 부여하지 않음
- 관리자가 검토 후 경고, 정지, 삭제 같은 운영 조치를 결정

숨김 해제 시에는 현재 신고 수를 검토 기준으로 저장해서 같은 누적 신고 수만으로 바로 다시 숨김 처리되지 않게 했습니다.

## Q7. Redis는 왜 사용했나요?

Redis는 Refresh Token 저장에 사용했습니다.

Redis를 사용한 이유는 다음과 같습니다.

- Refresh Token 만료 시간을 TTL로 자동 관리할 수 있음
- 로그아웃 시 저장된 Refresh Token을 삭제해서 재사용을 막을 수 있음
- 토큰 재발급 시 서버가 Refresh Token 유효성을 확인할 수 있음
- 추후 강제 로그아웃, 인기글 캐싱, 조회수 증가량 캐싱에도 확장 가능

MySQL은 영구 데이터 저장소이고, Redis는 빠른 만료/캐시/인증 보조 저장소로 역할을 분리했습니다.

## Q8. 인기글 기준은 무엇인가요?

현재 인기점수는 다음 공식으로 계산합니다.

```text
인기점수 = 조회수 + 댓글수 * 3 + 반응수 * 2
```

댓글은 사용자가 시간을 들여 참여한 행동이므로 가중치를 3으로 주고, 반응은 더 가벼운 참여 행동이므로 가중치를 2로 주었습니다.
신고로 블라인드 처리된 게시글은 인기글에서 제외합니다.

## Q9. 북마크는 어떻게 중복 저장을 막나요?

`bookmarks` 테이블에서 `user_id + post_id` 조합에 유니크 제약을 걸었습니다.

같은 사용자가 같은 게시글을 여러 번 저장할 수 없고, 이미 저장된 글에서 저장 버튼을 다시 누르면 북마크가 해제됩니다.

## Q10. 프로젝트에서 가장 중요한 설계 포인트는 무엇인가요?

익명성과 운영 가능성의 균형입니다.

일반 사용자에게는 익명 번호만 보여주지만, 서버는 작성자 ID를 저장해서 권한 검증, 신고 처리, 관리자 제재를 수행합니다.
이 구조 덕분에 익명 커뮤니티의 편안함과 실제 서비스 운영에 필요한 통제 기능을 동시에 만족시킬 수 있습니다.
